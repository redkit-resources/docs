    Основы Архитектура Поведения Behavior w2behtree w2beh w2steer w2animev

    Архитектура системы поведения¶

    Глоссарий

    • BT - Behavior Tree
    • WS - Witcher Script
    • SSM - Scriptable State Machine
    • FSM - Finite State Machine (Архитектурный паттерн)
    • NPC - Non Playable Character

    Сначала - три разных "behavior"¶

    Главный источник путаницы и его стоит проговорить сейчас, отдельно: слово "behavior" в движке означает три несвязанных механизма. Они соединены только через переменные и события, но живут на разных слоях.

    Scriptable State Machine - "в каком состоянии объект" (object-state layer)¶

    Кончечный автомат (FSM) на Witcher Script, состояние самого объекта, а не его анимация и не ИИ-цель. Общий механизм для всех, но у игрока он занимает особое место: раз у игрока нет Behavior Tree, именно SSM становится главным источником решений игрока. То есть у NPC решения делает BT, SSM просто отражает состояние, а у игрока SSM и решает, и отражает состояние одновременно.

    Behavior Tree - "что делать" (decision layer)¶

    Дерево принятия решений оно же дерево поведенией. Отвечает за ИИ действия. Есть только у NPC - у игрока такого слоя нет вообще, решения игрока идут напрямую из ввода и SSM игрока.

    Behavior Graph - "что играет" (presentation layer)¶

    Это не про решения вообще, это про то, как решение выглядит. В других движках это называется анимационный граф (animgraph). Анимационная state machine, которая смешивает позы скелета и выдает итоговую позу. Она общая для всех (NPC и игрок), но сама ничего не решает - только реагирует на переменные/события, которые пришли снаружи (от BT или от SSM).

    NPC vs Player. Общее исполнение, разное управление¶

    Общая исполнительная часть у всех одинаковая, разница только в управляющей части между игроком и NPC, у игрока нет BT поскольку это AI для NPC, решения игрока же дает ввод и глобальная SSM CR4Player. Она командует общей исполнительной частью напрямую и, в отличии от NPC замыкает цикл через animation events, для игрока это основной механизм тайминга действий, а не вспомогательный. О циклах событий будет в конце. У NPC состояния скорее обслуживают BT дерево.

    Scripting & state machines¶

    SSM на WS, паттерн на котором держится вся игровая логика. В краце, объект может быть в одном из заданных состояний, состояние это фактически цикл который крутится в отдельном ассинхроном потоке (latent функции), объект переключается по очереди между этими состояниями, например ходит -> атакует -> мертв.

    Ключевые классы

    • СActor
    • CGameplayEntity
    • CR4Player

    Постановщик (AI Orchestration)¶

    Заселение мира и расписание: Кто существует, где появляется и какой высокуровневой задачей занят. Создает NPC и настравивает его behavior tree. Игрока не трогает вообще.

    Ключевые классы:

    • CEncounter - Объект-постановвщик, spawn tree + пул существ
    • CCommunitySystem - Система спавна NPC по расписаниям
    • CJobTree - Рабочие точки и их анимации
    • CBehTreeReactionManager - Стимулы мира и их реации

    Behavior Trees¶

    Слой AI решений: Каждый тик обходит приоритетное дерево, выбирает действие и рассылает команды на исполнение. Сердце повередния NPC.

    Ключевые классы:

    • CBehTree (.w2behtree) - ассет дерева
    • IBehTreeTask - интерфейс для тасков на Witcher Script

    Movement¶

    Считает скорость и направление за кадр, держит агента на navmesh, согласует root motion с желаемым движением. У NPC им рулит steering граф (.w2steer), у игрока же ввод напрямую, по этому компонент общий, но драйверы разные.

    Ключевые классы:

    • CMovingAgentComponent - компонент движения
    • CMoveSteeringBehavior (.w2steer) - дерево steering
    • CMovementAdjustor - подгонка анимации в цель

    Behavior graphs¶

    Анимационная state machine и сеть смешивания поз: переменные и события приходят на вход со скриптов и деревьев, на выходе готовая поза скелета на каждый кадр.

    Ключевые классы:

    CBehaviorGraph (.w2beh) - ассет графа

    Animation System¶

    Считает позу в кости, извлекает root motion и в нужный момент шлет animation events. Держит граф + скелет.

    Ключевые классы:

    CAnimatedComponent - центр: стек граф + скелет CSkeleton - скелет CExtAnimEvent - запеченные в клип события

    Animation Events¶

    Необходимы для двусторонней связи: логика запускает анимацию, а запеченные в клип события в нужный кадр вызывают именованные OnAnimEvent_* функции обратно на объекте.

    AddAnimEventCallback('Имя','OnAnimEvent_X') - Регистрациия коллбека в скриптах

    Когда воспроизведение доходит до события - CAnimatedComponent зовет эту функцию по имени
